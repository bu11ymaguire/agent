SPN Shopping Agent + RA-Rec 데모 개선 피드백
==============================================

목표
----
현재 데모를 단순한 고정 시나리오 시각화에서 다음 인과관계를 실제로 확인할 수 있는 상태 기반 mock 데모로 개선한다.

사용자 발화
→ 자연어 선호 상태 변화
→ 검색 질의 및 리뷰 근거 변화
→ 상품 점수와 순위 변화
→ 최종 추천 설명 변화

이번 개선에서는 실제 LLM, 외부 쇼핑 API, 웹 크롤러를 붙이지 않아도 된다.
모든 동작은 로컬 mock 데이터와 결정론적 TypeScript 함수로 재현 가능해야 한다.
UI 언어와 기존의 SPN=에메랄드, RA-Rec=블루 계열 시각 체계는 유지한다.


1. 아키텍처 구조 변경
----------------------

현재처럼 SPN Agent 내부에 RA-Rec 전체를 하나의 중첩 서비스로만 표시하기보다,
RA-Rec을 “State Manager”와 “Recommendation Engine”으로 분리하고,
그 사이에 SPN Browsing Actions를 배치하는 파이프라인으로 변경한다.

권장 아키텍처:

사용자 발화
    ↓
SPN Shopping Agent (Task-level Orchestrator)
    ├─ 사용자 의도 분류
    ├─ SPN 5개 facet 추출
    ├─ 모호성 및 누락 정보 판단
    └─ 다음 행동 결정
           ├─ 정보 부족
           │     ↓
           │  SPN Ask User
           │     ↓
           │  사용자 응답 ↺
           │
           └─ 정보 충분
                 ↓
RA-Rec State Manager
    ├─ 기존 대화 상태 불러오기
    ├─ 새로운 선호·목적·제약 갱신
    ├─ 명시적 정보와 추론 정보 분리
    ├─ 거절·관심·수락·질의 기록
    ├─ 모순 및 우선순위 정리
    └─ 검색용 자연어 질의 생성
                 ↓
SPN Browsing Actions
    ├─ 상품 검색
    ├─ 상품 메타데이터 수집
    ├─ 리뷰 검색
    └─ 리뷰 이미지 및 배송·가격 정보 수집
                 ↓
RA-Rec Recommendation Engine
    ├─ 리뷰-사용자 의도 유사도 계산
    ├─ 상위 리뷰 점수의 상품 단위 집계
    ├─ 제약 충족도와 리뷰 근거를 결합한 순위화
    ├─ 추천 및 비교 결과 생성
    └─ 사용자 의도 기반 설명·QA 생성
                 ↓
SPN Response Composer
    ├─ RA-Rec 추천 결과
    ├─ 상품 이미지
    ├─ 가격·배송·재고 정보
    ├─ 근거 리뷰
    └─ 후속 행동 제안 통합
                 ↓
사용자 피드백
                 ↺ 상태 갱신

중요:
- SPN Shopping Agent는 파이프라인 전체를 통제하는 상위 오케스트레이터다.
- RA-Rec State Manager와 Recommendation Engine은 같은 RA-Rec 계열의 두 모듈이지만 책임을 분리한다.
- SPN Browsing Actions는 RA-Rec이 생성한 검색 질의를 받아 증거를 수집한다.
- RA-Rec이 사용자에게 직접 질문 문장을 생성하지 않도록 한다.
  RA-Rec은 필요한 정보와 이유를 구조화해서 반환하고, SPN Agent가 자연스러운 질문으로 표현한다.
- RA-Rec의 설명은 최종 사용자 응답이 아니다.
  SPN Response Composer가 가격, 배송, 이미지, 후속 행동을 결합해 최종 응답을 만든다.


2. 컴포넌트별 책임
------------------

SPN Shopping Agent
- 전체 쇼핑 세션 계획
- 사용자 의도 및 SPN 추출
- 모호성 계산과 다음 행동 선택
- 사용자 질문 표현
- 상품·리뷰·이미지 탐색 실행
- RA-Rec 결과와 브라우징 결과를 최종 응답으로 통합

RA-Rec State Manager
- 대화 상태의 단일 진실 공급원
- hard constraint, soft preference, subjective need, trade-off 관리
- explicit / implicit / inferred 정보 구분
- 관심, 비교, 거절, 선택 상태 기록
- 이전 발화와 새 발화의 모순 처리
- 검색용 자연어 query 생성

RA-Rec Recommendation Engine
- 리뷰 관련도 계산
- late-fusion 방식의 상품 점수 집계
- 상품 순위화
- 추천 이유와 비교 근거 생성
- 추천 상품에 대한 QA

SPN Browsing Actions
- 로컬 mock 상품·리뷰 데이터 검색
- 상품 상세, 이미지, 가격, 배송 정보 조회
- RA-Rec 검색 질의에 맞는 증거 반환

책임 중복을 피한다.
특히 “추가 질문 생성”, “리뷰 검색”, “최종 응답 생성”을 SPN과 RA-Rec 양쪽에서 중복 수행하지 않는다.


3. 상태 및 데이터 모델 개선
----------------------------

3-1. SPN을 문자열 태그 배열이 아니라 5개 facet 구조로 변경한다.

type SPNFacetName =
  | "subjective_property"
  | "event"
  | "activity"
  | "goal_purpose"
  | "goal_audience";

type EvidenceOrigin = "explicit" | "implicit" | "inferred";
type EvidenceStatus = "confirmed" | "unconfirmed" | "superseded";

type PreferenceValue = {
  id: string;
  valueText: string;
  origin: EvidenceOrigin;
  confidence: number;       // 0~1
  status: EvidenceStatus;
  evidenceTurnIds: string[];
  updatedAtTurnId: string;
};

type SPNState = Record<SPNFacetName, PreferenceValue | null>;

facet이 확인되지 않았으면 억지로 채우지 말고 null을 유지한다.


3-2. RA-Rec Dialogue State를 쇼핑 퍼널에 맞게 확장한다.

type DialogueState = {
  category?: PreferenceValue;

  hardConstraints: Record<string, PreferenceValue>;
  softConstraints: Record<string, PreferenceValue>;
  subjectiveNeeds: SPNState;
  tradeoffs: PreferenceValue[];

  recommendedItems: string[];
  shortlistedItems: string[];
  inspectedItems: string[];
  rejectedItems: Array<{
    productId: string;
    reason: PreferenceValue;
  }>;
  cartedItems: string[];
  purchasedItems: string[];

  currentItem?: string;
  unresolvedPreferences: Array<{
    field: string;
    reason: string;
    priority: number;
  }>;

  preferenceHistory: Array<{
    turnId: string;
    changedPaths: string[];
  }>;
};

“첫 번째 제품을 자세히 볼게요”는 accepted/purchased가 아니다.
해당 상품을 inspectedItems 또는 shortlistedItems에 추가한다.
acceptedItems라는 표현은 제거하거나, 실제 최종 선택 의미로만 제한한다.


3-3. 리뷰를 상품과 직접 연결하고 점수 근거를 보존한다.

type Review = {
  id: string;
  productId: string;
  text: string;
  helpfulness?: number;
  source: "mock";
};

type RankedReview = Review & {
  similarityScore: number;
  preferenceCoverageScore: number;
  reliabilityScore: number;
  totalScore: number;
  matchedPreferenceIds: string[];
};

현재처럼 retrievedReviews를 string[]로 관리하지 않는다.
화면에 표시되는 리뷰와 실제 상품 점수에 사용된 리뷰가 동일해야 한다.


3-4. 상품 데이터에 리뷰 수를 추가한다.

type Product = {
  ...
  rating: number;
  reviewCount: number;
  reviews: Review[];
};

현재 시나리오가 “리뷰 수가 많은 상품”을 핵심 기준으로 사용하므로,
reviewCount가 실제 데이터에 반드시 존재해야 한다.
값은 mock임을 명시하며 실제 판매 데이터처럼 표현하지 않는다.


3-5. 추천 점수의 세부 항목을 저장한다.

type ScoreBreakdown = {
  hardConstraintMatch: number;
  metadataMatch: number;
  subjectiveNeedMatch: number;
  reviewEvidenceScore: number;
  evidenceReliability: number;
  total: number;
};

type RankedProduct = {
  productId: string;
  rank: number;
  score: ScoreBreakdown;
  evidenceReviewIds: string[];
  matchedPreferenceIds: string[];
};


4. 우선 구현해야 할 핵심 기능(P0)
---------------------------------

P0-1. 하드코딩된 점수와 후보 ID 제거
- 상품 ID에 따라 94점, 87점 등을 직접 지정하지 않는다.
- candidateProductIds를 시나리오에 미리 넣고 그대로 출력하지 않는다.
- 현재 DialogueState와 Product/Review 데이터를 입력으로 받아 rankProducts()가 후보와 점수를 계산해야 한다.
- 같은 입력에는 항상 같은 결과가 나오는 결정론적 함수로 구현한다.

권장 mock 점수식:

reviewTotal
= 0.65 * intentSimilarity
+ 0.25 * preferenceCoverage
+ 0.10 * reliability

itemTotal
= 0.30 * hardConstraintMatch
+ 0.20 * metadataMatch
+ 0.25 * subjectiveNeedMatch
+ 0.20 * averageTopKReviewScore
+ 0.05 * evidenceReliability

- 모든 점수는 0~100으로 정규화한다.
- topK는 기본 3으로 둔다.
- reviewCount는 log scaling하여 evidenceReliability에 일부 반영하되,
  인기 상품이 무조건 상위에 오르지 않도록 비중을 낮게 유지한다.
- 실제 embedding이 없으므로 mock 단계에서는 한국어 토큰/키워드 overlap과
  동의어 사전을 이용한 deterministic similarity를 사용해도 된다.


P0-2. 사용자 입력이 실제 상태를 바꾸게 한다
현재 입력 내용과 관계없이 다음 고정 턴으로 이동하는 동작을 제거한다.
최소한 다음 발화 유형을 처리한다.

- 예산:
  “20만 원 이하였으면 좋겠어요.”
  → hardConstraints.budget 갱신

- 사용 목적:
  “필기를 많이 할 것 같아요.”
  → subjectiveNeeds.activity 또는 softConstraints.purpose 갱신
  → 펜/필기 관련 상품 가중치 증가

- 리뷰 선호:
  “후기가 많은 게 믿음이 가요.”
  → review_signal 갱신

- 상세 보기:
  “첫 번째 제품을 자세히 볼게요.”
  → inspectedItems, shortlistedItems, currentItem 갱신

- 거절:
  “첫 번째 제품은 저장 공간이 너무 작아요.”
  → rejectedItems에 상품과 거절 이유 추가
  → storage 선호/제약 추출
  → 즉시 재순위화

- 비교:
  “첫 번째와 두 번째를 비교해줘.”
  → 두 상품의 차이와 관련 리뷰 근거 반환

실제 LLM 대신 정규식, 키워드 규칙, scenario branch를 조합해도 된다.
단, 자유 입력이 최소한 위 상태 전환 중 하나를 실제로 발생시켜야 한다.


P0-3. 명시적 정보와 추론을 분리한다
예:
사용자 발화: “후기가 많은 게 믿음이 가요.”

직접 확인된 정보:
- review_signal = “리뷰 수를 중요하게 봄”
- origin = explicit
- confidence = 1.0

추론 정보:
- risk_attitude = “구매 실패 위험을 낮추고 싶을 가능성”
- origin = inferred
- confidence < 1.0
- status = unconfirmed

추론값을 확정 사실처럼 hard/soft constraint에 바로 넣지 않는다.
UI에서 explicit, implicit, inferred badge를 서로 다르게 표시한다.


P0-4. 거절 이유가 추천 순위를 바꾸는 시나리오 추가
반드시 다음 장면을 시연할 수 있어야 한다.

1) 저가·리뷰 수 기준으로 64GB 상품이 1위
2) 사용자: “저장 공간이 너무 작아요.”
3) rejectedItems에 상품과 이유가 기록됨
4) “저장 공간을 더 중요하게 봄”이라는 선호가 상태에 추가됨
5) 128GB 또는 256GB 상품의 순위가 상승
6) 변경 전/후 점수 차이와 근거 리뷰가 화면에 표시됨

이 장면이 데모의 핵심이다.
“사용자의 한 발화가 상태, 근거, 점수, 순위를 모두 바꾼다”는 것을 보여줘야 한다.


P0-5. 모호성 점수에 근거를 제공한다
92, 84, 46 같은 숫자를 배열에 직접 저장하지 않는다.

type VaguenessBreakdown = {
  categoryBreadth: number;
  missingRequiredInfo: number;
  unresolvedSPN: number;
  contradictionPenalty: number;
  total: number;
  threshold: number;
  reasons: string[];
};

UI 예시:
모호성 69%
- 예산이 확인되지 않음: +20
- 사용 목적이 확인되지 않음: +25
- 상품 범위가 넓음: +18
- 리뷰 선호가 확인됨: -8

total이 threshold보다 높으면 SPN Agent가 Ask User를 선택한다.
질문은 누락 정보 중 예상 순위 변화를 가장 크게 만드는 항목을 우선한다.


P0-6. RA-Rec 결과와 SPN 최종 응답을 분리한다
RA-Rec RecommendationResponse 예시:

{
  "updatedDialogueState": {},
  "rankedProducts": [],
  "reviewEvidence": [],
  "explanation": "추천 판단용 설명",
  "unresolvedPreferences": []
}

SPN Final Response 예시:

{
  "message": "사용자에게 보여줄 최종 문장",
  "productCards": [],
  "priceEvidence": [],
  "deliveryEvidence": [],
  "imageEvidence": [],
  "reviewEvidence": [],
  "nextActions": []
}

RA-Rec explanation과 사용자 채팅 메시지에 같은 문자열을 재사용하지 않는다.
SPN Response Composer가 RA-Rec 결과에 가격, 이미지, 배송, 후속 행동을 추가해야 한다.


P0-7. agentSnapshots 분리 문제 해결
app/page.tsx의 agentSnapshots 배열과 data/scenario.json의 turns 배열을
인덱스로 묶지 않는다.

다음 중 하나로 변경한다.
A. 각 scenario turn 안에 SPN state와 vagueness breakdown을 포함
B. 현재 DialogueState에서 SPN snapshot을 함수로 계산

권장: B 방식.
deriveSPNSnapshot(dialogueState, currentInput, products)와
computeVagueness(dialogueState)를 순수 함수로 구현한다.


5. UI 개선
-----------

중앙 패널
- SPN Orchestrator
- 조건에 따른 Ask User 분기
- RA-Rec State Manager
- SPN Browsing Actions
- RA-Rec Recommendation Engine
- SPN Response Composer

위 단계를 순서대로 표시한다.
단순히 RA-Rec 하나를 SPN 컨테이너 안에 넣는 구조보다,
모듈 사이의 요청·응답과 데이터 흐름이 보이게 한다.
SPN이 전체 흐름을 통제한다는 라벨 또는 외곽선을 유지한다.

우측 패널
1. RA-Rec Dialogue State
2. 이번 턴 State Diff
3. SPN 5개 facet
4. explicit / implicit / inferred 상태
5. 검색용 자연어 query
6. 상품별 Score Breakdown
7. 상품별 Top-k Review Evidence
8. 이전 순위와 현재 순위 변화

상품 카드
- 상품명
- 가격
- 평점
- reviewCount
- total score
- score breakdown
- 추천에 사용된 리뷰 근거
- 어떤 선호와 매칭되었는지 표시
- mock data badge 표시

대화 패널
- 기존 suggestion 버튼은 유지
- 자유 입력도 처리
- 현재 선택/관심/거절 상태를 정확한 용어로 표시
- “자세히 보기”와 “구매 결정”을 구분

7. 필수 데모 시나리오
---------------------

Scenario A: 모호한 첫 구매 요청
- “태블릿을 처음 사는데 뭘 봐야 할지 모르겠어요.”
- SPN 추출
- 모호성 근거 표시
- SPN Agent가 추가 질문 선택

Scenario B: 목적에 따른 순위 변화
- “주로 필기를 할 것 같아요.”
- activity/purpose 갱신
- 필기 관련 메타데이터와 리뷰가 높은 상품 상승

Scenario C: 거절 이유에 따른 재추천
- “첫 번째는 저장 공간이 너무 작아요.”
- rejectedItems 및 reason 기록
- storage 선호 갱신
- 추천 순위 재계산
- 이전/현재 순위 비교

Scenario D: 상세 보기와 최종 선택 구분
- “첫 번째를 자세히 볼게요.”
  → inspected/shortlisted
- “이 제품으로 살게요.”
  → purchased 또는 최종 선택 상태
