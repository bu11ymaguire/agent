# 실제 데모 상호작용 시나리오

이 문서는 현재 실행 경로인 `app/page.tsx`와 `lib/*`의 동작을 기준으로 작성한다. `data/scenario.json`, `app.js` 등 이전 프로토타입 파일은 현재 화면의 대화 진행에 사용되지 않는다.

## 한눈에 보는 실제 흐름

전송 버튼은 사용자가 직접 문장을 입력받는 UI가 아니라, 현재 상태에 맞는 다음 고정 데모 발화를 `processInput`에 전달한다. 이 데모의 실제 순서는 다음과 같다.

1. **첫 구매·20만 원 이하 요청** → 즉시 첫 추천
2. **첫 번째 제품의 저장공간 거절** → 거절 제품을 제외하고 저장공간을 반영해 재추천
3. **새 첫 번째 제품 상세 보기** → 관심/검토 상태를 기록한 뒤 같은 추천 파이프라인을 다시 실행
4. **현재 제품 구매** → 구매 상태를 기록한 뒤 같은 추천 파이프라인을 다시 실행

따라서 화면의 네 번의 사용자 발화가 네 개의 턴이다. 상세 보기와 구매도 `recommend` 정책과 랭킹 파이프라인을 유지하지만, 응답 생성 단계는 상품 행동에 맞는 대사를 선택한다. 상세 보기에는 현재 제품의 핵심 사양을, 구매에는 구매 의사 기록과 목업 결제 안내를 보여 준다.

| 턴 | 사용자 발화 | 해석된 상품 행동 | State Diff의 핵심 | 화면상 결과 |
| --- | --- | --- | --- | --- |
| 1 | “태블릿을 처음 사는데… 20만 원 이하…” | 없음 | 카테고리·예산·첫 구매자 | 안드로이드 태블릿 10인치 4GB+64GB 우선 추천 |
| 2 | “첫 번째 제품은 저장 공간이 너무 작아요.” | 현재 1위 거절 | `rejectedItems`, `storagePriority` | TECLAST T40 10.4인치 128GB 우선 추천 |
| 3 | “첫 번째 제품을 자세히 볼게요.” | 현재 1위 상세 보기 | `inspectedItems`, `shortlistedItems`, `currentItem` | T40의 화면·메모리·저장공간을 포함한 상세 확인 응답 |
| 4 | “이 제품으로 살게요.” | `currentItem` 구매 | `purchasedItems` | 구매 의사 기록과 목업 결제 안내 응답 |

## 공통 처리 순서

각 턴은 아래 순서로 처리된다.

```text
사용자 발화
  → 이전 상태로 기존 순위 계산
  → 입력 파싱 및 DialogueState 갱신 / State Diff 생성
  → SPN 모호성 계산과 정책 선택
  → (recommend일 때) 질의 생성 → mock 카탈로그 탐색 → 상품·리뷰 순위 계산
  → recommendedItems 갱신 → 최종 응답 생성
```

중요한 기준은 다음과 같다.

- `첫 번째 제품`은 고정된 상품명이 아니라 **해당 턴 직전 순위의 1위**다. 그래서 2턴에서는 안드로이드 태블릿이, 3턴에서는 재추천된 T40이 대상이 된다.
- `State Diff`는 `updateDialogueState`가 인식한 변경 경로다. 추천 뒤에 채워지는 `recommendedItems`는 화면의 Diff에는 포함되지 않는다.
- 추천은 `browseMockCatalog`의 예산·카테고리 결과를 바탕으로 한다. 최종 상품 랭킹은 `rankProducts`가 다시 계산하며, `rejectedItems`에 있는 상품은 여기서 제외된다.
- 다만 탐색 증거(`priceEvidence`, `deliveryEvidence`) 자체는 거절 상품을 별도로 제외하지 않는다. 따라서 그 증거 목록에는 이전에 거절한 상품이 남을 수 있지만, 추천 카드의 순위 후보에는 남지 않는다.

## 1턴 — 첫 구매와 예산이 있는 첫 요청

### 입력

> 태블릿을 처음 사는데 뭘 보고 골라야 할지 모르겠어요. 20만 원 이하였으면 좋겠어요.

### 상태 변화

`parseUserInput`은 태블릿 카테고리, 200,000원 예산, 첫 구매 맥락을 추출한다. `updateDialogueState`는 다음을 저장한다.

- `category`: 태블릿
- `hardConstraints.budget`: 200,000원 이하
- `subjectiveNeeds.goal_audience`: 첫 태블릿 구매
- `softConstraints.buyerExperience`: 첫 태블릿 구매

이 값들은 모두 현재 턴을 근거(`evidenceTurnIds`, `updatedAtTurnId`)로 갖는 명시적(`explicit`), 확정(`confirmed`) 값이다.

### 왜 추가 질문이 아니라 추천인가

용도(`activity`)와 주관적 구매 기준(`subjective_property`)은 비어 있다. 그러나 모호성 점수는 용도 미확인 25점과 주관적 기준 미확인 10점으로 합계 **35점**이다. 정책의 질문 임계값은 45점이므로 `selectAction`은 `ask_user`가 아니라 `recommend`를 선택한다.

즉, 문서에 있던 “첫 턴에서 용도를 묻는다”는 설명은 현재 코드와 다르다. 첫 턴은 곧바로 검색·랭킹·추천까지 진행한다. 20만 원 이하의 태블릿 중 리뷰·가격 관련 점수를 계산해 안드로이드 태블릿 10인치 4GB+64GB가 1위가 되고, 화면에는 가격·배송 조건과 리뷰 근거를 함께 보라는 일반 추천 응답이 표시된다.

참고로 “첫 구매이므로 실패 위험을 낮추려 할 수 있다”는 숨은 의도 가설은 별도로 표시되지만, 이는 `inferred / unconfirmed` 가설이며 랭킹 제약 자체는 아니다.

## 2턴 — 저장공간 이유의 거절과 재추천

### 입력

> 첫 번째 제품은 저장 공간이 너무 작아요.

### 상태 변화

이 발화의 `첫 번째 제품`은 1턴 직전/결과 순위에서 1위였던 `android-tablet-10`이다. 상태 관리자는 다음을 갱신한다.

- `rejectedItems`: `android-tablet-10`과 거절 이유 “저장 공간이 너무 작음”
- `softConstraints.storagePriority`: 128GB 이상 저장공간 선호

따라서 화면의 State Diff에는 `rejectedItems`와 `softConstraints.storagePriority`가 나타난다.

### 순위 변화

저장공간 선호는 메타데이터 적합도와 주관적 요구 적합도에 저장용량 가점을 더한다. 동시에 `rankProducts`는 거절된 안드로이드 태블릿을 후보에서 제외한다. 예산 안에서 128GB인 TECLAST T40 10.4인치 128GB가 새 1위가 되어, 화면과 같이 199,000원·무료 배송 조건의 추천으로 바뀐다.

여전히 용도와 주관적 기준은 비어 있지만 모호성은 35점이어서 정책은 계속 `recommend`다. 거절은 질문을 유도하지 않고 즉시 재추천을 유도한다.

## 3턴 — 상세 보기의 상태 기록

### 입력

> 첫 번째 제품을 자세히 볼게요.

### 상태 변화

이 시점의 1위는 T40이므로 `inspect_first`는 T40을 대상으로 동작한다.

- `inspectedItems`에 T40 추가
- `shortlistedItems`에 T40 추가
- `currentItem`을 T40으로 설정

각 배열은 중복 없이 유지된다. 따라서 이 턴은 “추천 수락”이나 구매 확정이 아니라 관심 제품을 상세 검토 대상으로 기록하는 행동이다.

### 실제 응답 동작

정책은 여전히 `recommend`이므로 저장공간 선호와 기존 거절을 반영한 추천·랭킹은 다시 실행된다. 다만 응답 생성 단계가 `inspect_first`를 인식해 T40의 화면 크기, 메모리, 저장공간과 제품 설명을 포함한 상세 확인 대사를 만든다.

## 4턴 — 구매 상태 기록

### 입력

> 이 제품으로 살게요.

### 상태 변화

`purchase_current`는 `currentItem`이 있어야 동작한다. 3턴에서 현재 관심 제품이 T40으로 설정되었으므로, 이 턴에서 다음 상태가 추가된다.

- `purchasedItems`: T40

`currentItem`, 검토 이력, 거절 이력은 그대로 보존된다. 구매 완료 여부는 `purchasedItems`로 확인할 수 있다.

### 실제 응답 동작

정책은 여전히 `recommend`이고 구매 이력은 랭킹 공식에 가점·감점을 주지 않는다. 따라서 T40이 계속 1위로 계산되며, 응답 생성 단계는 `purchase_current`를 인식해 구매 의사를 기록했음을 알린다. 실제 결제를 수행하지 않는 목업임도 함께 안내한다.

## 주변 상태와 화면의 관계

| 상태 | 갱신 턴 | 다음 추천에 미치는 영향 | 화면에서 확인할 위치 |
| --- | --- | --- | --- |
| 예산 | 1 | 카탈로그 탐색에서 20만 원 이하 상품을 우선 노출 | 현재 이해한 사용자 요구, State Diff, JSON |
| 첫 구매자 | 1 | 숨은 의도 가설의 근거가 됨 | 사용자 요구, 숨은 의도, JSON |
| 거절 상품 | 2 | 최종 랭킹 후보에서 제외 | State Diff, JSON, 추천 카드 |
| 저장공간 선호 | 2 | 저장용량이 큰 상품의 적합도 점수를 높임 | 사용자 요구, State Diff, 점수 상세 |
| 상세 보기/관심 상품 | 3 | `currentItem`을 정해 구매 발화의 대상을 만듦 | State Diff, JSON |
| 구매 상품 | 4 | 구매 이력으로만 보존되며 현 랭킹에는 영향 없음 | State Diff, JSON |

## 발표용 요약

이 데모는 사용자의 발화를 상태로 누적하고, 거절 이유를 다음 추천의 제약으로 바꾸는 흐름을 보여 준다. 첫 턴은 용도를 묻는 단계가 아니라 예산과 첫 구매 정보만으로 바로 추천하는 단계다. 이후 저장공간 거절은 실제로 기존 1위를 제외하고 T40을 새 1위로 만든다. 상세 보기와 구매도 랭킹은 계속 계산하지만, 각 상품 행동에 맞는 상세 확인·구매 의사 기록 대사를 표시한다.
